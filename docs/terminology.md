# Терминология и именование

Статус: active
Роль: repo-level SSOT для терминов и именования документации
Последнее обновление: 2026-03-15

## Назначение

Этот документ фиксирует repo-wide значения терминов, которые регулярно
используются в документации, SSOT, feature-docs, ADR и runtime-описаниях.

Цель:

- сократить размытость между близкими словами;
- стабилизировать названия документов;
- не допускать случайного смешения `flow`, `process`, `workflow`,
  `lifecycle` и `pipeline`.

Этот документ описывает именно правила терминологии и именования.

Он не заменяет:

- feature-спеки;
- профильные SSOT по конкретным flow;
- ADR по архитектурным решениям.

## Канонические термины

### `flow`

Основной термин для agent-driven или operator-driven сценария, который проходит
через шаги, статусы, переходы и human gates.

Использовать, когда важно ответить на вопрос:

- как сценарий проходит от входа к выходу;
- какие шаги и переходы разрешены;
- где находятся точки вмешательства человека.

Типичные примеры:

- `issue-analysis-flow`;
- `issue-implementation-flow`.

Правило для этого репозитория:

- `flow` является термином по умолчанию для канонических SSOT, описывающих
  выполнение issue-level или stage-level сценария.

### `lifecycle`

Термин для жизненного цикла сущности во времени.

Использовать, когда важно ответить на вопрос:

- в каких состояниях живет объект;
- какие terminal и intermediate states у него есть;
- как меняется одна сущность, а не маршрут выполнения работы.

Типичные примеры:

- issue lifecycle;
- branch/worktree/PR lifecycle;
- session lifecycle.

Правило:

- `lifecycle` не должен подменять `flow`, если документ описывает именно
  порядок выполнения сценария, а не жизнь сущности.

### `process`

Термин для устойчивого повторяемого способа работы на уровне репозитория,
команды или функции.

Использовать, когда важно ответить на вопрос:

- как в целом организована область работы;
- какие правила, роли, обязательные шаги и точки синхронизации существуют;
- как выглядит постоянный operating model, а не один сценарий.

Типичные примеры:

- documentation process;
- incident management process.

Правило:

- `process` использовать для governance- и operating-model документов, а не
  для stage-specific issue routing.

### `workflow`

Более общий термин для формализованного порядка работы с ролями, handoff и
правилами.

Использовать, когда нужно объяснить общую картину в prose или сослаться на
широкий класс рабочих маршрутов без излишней терминологической жесткости.

Правило:

- в канонических названиях новых документов `workflow` не является термином по
  умолчанию;
- если `flow`, `process`, `lifecycle` или `pipeline` описывают контракт
  точнее, нужно использовать их;
- `workflow` допустим в explanatory тексте, summary-слоях и общих формулировках.

### `pipeline`

Термин для линейной или почти линейной цепочки автоматической обработки
артефакта.

Использовать, когда важно ответить на вопрос:

- через какие технические стадии проходит артефакт;
- какая последовательная обработка применяется автоматически.

Типичные примеры:

- CI pipeline;
- build pipeline;
- ingestion pipeline.

Правило:

- `pipeline` не использовать как синоним `flow`, если речь идет об issue-level
  orchestration, human gates или stage dispatch.

## Практическая иерархия

Рабочая эвристика для этого репозитория:

- `process` шире и описывает устойчивый способ работы;
- внутри `process` могут существовать отдельные `flow`;
- `pipeline` является частным случаем `flow` с упором на последовательную
  автоматическую обработку;
- `lifecycle` стоит отдельно и описывает жизнь сущности, а не маршрут работы;
- `workflow` остается допустимым общим словом, но не preferred label для новых
  канонических документов.

## Правила именования документов

Использовать следующие суффиксы, когда документ сам является каноническим
носителем соответствующего контракта:

- `*-flow.md` — сценарий выполнения с шагами, статусами, переходами и human
  gates;
- `*-lifecycle.md` — жизненный цикл сущности;
- `*-process.md` — устойчивый repo-level или team-level порядок работы;
- `*-pipeline.md` — линейная техническая цепочка обработки;
- `workflow` в именах новых документов не использовать как default, если есть
  более точный термин.

Примеры для текущего репозитория:

- [issue-analysis-flow.md](./issue-analysis-flow.md);
- [issue-implementation-flow.md](./issue-implementation-flow.md);
- [documentation-process.md](./documentation-process.md).

## Правила применения

При создании нового документа, SSOT или feature-артефакта автор должен сначала
выбрать, что именно описывается:

1. сценарий выполнения работы;
2. жизненный цикл сущности;
3. устойчивый организационный порядок;
4. техническая последовательная обработка;
5. общий explanatory контекст без строгого канонического ярлыка.

После этого выбирается термин:

- `flow` для пункта 1;
- `lifecycle` для пункта 2;
- `process` для пункта 3;
- `pipeline` для пункта 4;
- `workflow` только для пункта 5 или для prose, где намеренно не нужен более
  узкий термин.

Если документ описывает несколько осей сразу, заголовок и путь должны
отражать главный контракт, а остальные аспекты раскрываются внутри текста.
