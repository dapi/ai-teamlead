# Feature 0002: Что строим

## Проблема

Сейчас у `ai-teamlead` нет формализованного способа подключить новый
репозиторий к системе. Из-за этого repo-local файлы, которые должны задавать
project-specific контракт, начинают появляться как неоформленные технические
заготовки.

Нужна отдельная точка входа, которая:

- создает versioned project-local слой
- не смешивает versioned contract с runtime state
- дает владельцу репозитория стартовый набор файлов для настройки flow

## Пользователь

Основной пользователь:

- владелец репозитория
- разработчик, который подключает `ai-teamlead` к своему проекту

## User Story

Как владелец или разработчик репозитория, я хочу запустить
`ai-teamlead init` в чистом git-репозитории с настроенным `origin`, чтобы
сразу получить готовый versioned project-local contract layer без ручной
сборки `./.ai-teamlead/` и связанных файлов.

## Use Case

Базовый пользовательский сценарий первой версии:

1. У пользователя есть чистый git-репозиторий без `./.ai-teamlead/`.
2. В репозитории уже настроен `origin`, указывающий на GitHub-репозиторий.
3. Пользователь запускает `ai-teamlead init` из корня репозитория.
4. Команда создает `./.ai-teamlead/` и весь минимальный набор versioned
   project-local файлов.
5. Если в корне еще нет `./init.sh`, команда создает симлинк
   `./init.sh -> ./.ai-teamlead/init.sh`.
6. Пользователь получает готовую стартовую структуру, которую можно
   закоммитить, ревьюить и донастроить под проект.

## Результат

Полезным результатом первой версии считается команда:

- `ai-teamlead init`

которая в корне git-репозитория создает versioned project contract layer:

- `./.ai-teamlead/settings.yml`
- `./.ai-teamlead/README.md`
- `./.ai-teamlead/init.sh`
- `./.ai-teamlead/launch-agent.sh`
- `./.ai-teamlead/flows/issue-analysis-flow.md`
- `./.ai-teamlead/flows/issue-analysis/`
- `./.claude/README.md`
- `./.codex/README.md`

После этого репозиторий получает минимальный project-local контракт, который
можно коммитить, ревьюить и настраивать под себя.

Если в корне репозитория отсутствует `./init.sh`, дополнительно создается
симлинк:

- `./init.sh -> ./.ai-teamlead/init.sh`

## Scope

В первую версию входит:

- команда `ai-teamlead init`
- создание каталога `./.ai-teamlead/`
- создание минимального набора versioned файлов
- создание root-level симлинка `./init.sh`, если он отсутствует
- идемпотентное поведение без перезаписи существующих файлов
- вывод результата инициализации в stdout

## Вне scope

- создание runtime-артефактов в `.git/.ai-teamlead/`
- автоматическое определение и запись `github.project_id`
- интерактивный wizard
- валидация пользовательских правок flow-документов
- создание дополнительных flow, кроме `issue-analysis-flow`

## Ограничения и предпосылки

- `init` работает только внутри git-репозитория
- `init` требует настроенный `origin`, указывающий на GitHub-репозиторий
- `init` создает только versioned project-local файлы
- `init` может дополнительно создать симлинк `./init.sh`, если в корне
  репозитория еще нет собственного `init.sh`
- runtime state продолжает жить отдельно в `.git/.ai-teamlead/`
- итоговые файлы должны быть пригодны для ручной донастройки владельцем
  репозитория
